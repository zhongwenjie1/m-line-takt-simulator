# M-Line 混流节拍仿真系统 改善计划方案 v2.9

> 更新日期：2026-06-12  
> 依据：《真实数据全风险调查报告（整合版）》与《真实数据可信度复核：提前加工与同槽占位》  
> 状态：已按全量真实数据二次复核更新，按“一轮只做一个任务”逐项实施

## 1. 改善目标

本计划用于修正真实数据调查中确认的排程、统计口径和界面表达风险。

整个实施过程遵守以下原则：

1. 一轮只处理一个明确问题。
2. 不顺手修改本轮范围之外的逻辑。
3. 每轮修改后先验证，不立即进入下一轮。
4. 每轮都使用真实数据回归，不能只验证小型示例。
5. 用户确认本轮结果后，再单独提交并进入下一轮。
6. 排程层记录真实时间，分析层判断业务含义，界面层只负责显示。
7. 总完成时间始终以最后一台目标车辆的下线完成时间为准。
8. 累计阻塞不参与总完成时间加法。
9. “程序输出与调查资料一致”只表示结果可复现，不表示模型符合真实产线物理规则。
10. 在有限车位、汇流服务顺序和投车规则完成修正与确认前，当前下线数、整体节拍和累计阻塞只作为修改前基线，不作为真实产线结论。

## 2. 新旧逻辑与回退方式

正式程序不增加“新逻辑/旧逻辑”切换开关，也不长期保留两套业务算法。

每轮通过以下方式对比和回退：

1. 修改前保存真实数据基线。
2. 每轮只产生一个独立提交。
3. 修改后生成新旧结果差异表。
4. 验证失败时通过 Git 回退本轮提交。
5. 验证通过后，正式程序只保留确认后的逻辑。

这样可以避免使用者误选旧算法，也避免程序长期维护两套互相分叉的逻辑。

## 4. 每轮固定实施流程

1. 确认当前仓库、分支、HEAD和工作区状态。
2. 在修改前记录本轮真实数据基线。
3. 只打开和修改本轮允许文件。
4. 执行针对本轮问题的最小场景验证。
5. 执行两组完整真实数据回归。
6. 执行语法检查。
7. 检查 Git diff，确认没有无关文件和无关逻辑。
8. 将过程和结果追加到 `ticket_export_app/docs/改善实施过程_v2.9.md`。
9. UI任务生成Mac测试包供人工点检；核心任务原则上不在同一轮打包。
10. 暂停，等待用户确认。
11. 用户确认后单独提交本轮修改。
12. 提交完成后才进入下一轮。

## 6. 第1轮：R2 0工时节点有限容量

### 问题

当前0工时节点不占车位，可以同时停放多台车。上游工位会过早放车，后车也可能过早开始加工。

全量真实数据已确认该问题不是局部截图现象：按比例场景有1357组同槽重叠，按数量场景有1624组；其中“零工时等待车尚未离开，后一台已开始加工”分别为986组和1086组。

### 目标

每个工位节点，包括0工时节点，每条实际流转线同一时刻最多容纳1台车。

### 实施规则

1. 0工时节点仍保留工位过程记录。
2. 0工时节点不占加工设备。
3. 0工时节点不强制车辆切换线别。
4. 0工时节点不参与工位能力判断。
5. 0工时节点拥有固定1台的流转车位容量。
6. 车位未释放时，上游车辆不能进入，并逐级向上游回堵。
7. 本轮不新增容量输入和UI配置。
8. 本轮不处理共用设备按实际到达顺序服务的问题。

### 主要修改文件

- `ticket_export_app/core/tickets.py`

### 最小场景

A工位单设备30秒 → 0工时节点 → B工位单设备100秒，共4台车。

修复后：

- 0工时节点任一时刻最多1台车；
- 第4台车在A工位的开始时间应为130秒，而不是当前的90秒。

### 真实数据验收

1. 1340秒时，Car#12与Car#14不能再同时占用ARHUD+慢充2号线；Car#16不能因该车位未计容量而提前进入ADAS。
2. Car#16 当前1325.5秒开始ADAS；按单槽容量约束，其开始时间不得早于占位链允许的1342.5秒，具体修复值必须由新排程完整重算确认。
3. 按数量场景中，Car#372、Car#375和Car#377当前曾同时出现在“气囊安装+绝缘”2号线；修复后同一时刻最多1台。
4. 任意工位动画槽位同时在位车辆数不得超过1，包括“0工时等待车+加工车”和“多台0工时等待车”。
5. 按比例场景同槽重叠由1357组降为0；按数量场景由1624组降为0。
6. “零工时等待车未离开、后一台开始加工”两组场景均降为0。
7. 修复后，工位卡片内部的橙色“还有X台”不应再作为正常业务状态出现。
8. 如果工位卡片仍出现“还有X台”，判定为容量、动画分组或线别映射仍有问题，继续排查，不用分类文案掩盖。
9. 同一加工设备不能出现超过有效设备能力的加工重叠。
10. 两组完整真实数据必须跑通，并输出逐车时间变化、下线数、整体节拍和等待变化；旧1046/1036台与65.2秒不得作为强制不变值。

## 7. 第2轮：R9 按比例投车平准化排序

### 问题

当前按比例模式把一个比例块直接展开。例如2:3:5生成 `AABBBCCCCC`，最大连续台数在比例模式中没有真正生效。

该顺序会直接改变工位到达波形、等待位置、窗口下线数和整体完成时间，因此不再归为单纯体验优化。本轮纳入当前改善范围，但仍保持“一轮一个任务”，不与R2或R10合并修改。

### 目标

在保持最终车型比例准确的同时，让任意前缀的车型分布尽量接近目标比例，并严格限制连续同车型数量。

### 排序规则

1. 使用“累计比例偏差最小”规则逐台选择车型。
2. 每一步优先选择当前实际累计占比最落后于目标占比的车型。
3. 最大连续台数作为硬约束：只要其他车型仍有可投配额，就不得超过设定值。
4. 多个车型条件相同时，按A→B→C固定顺序选择。
5. 同样输入必须始终生成相同序列。
6. 全量车型数量必须与理论比例分配结果完全一致。
7. 本轮只修改投车序列生成，不修改工位调度、等待和UI结果口径。

### 主要修改文件

- `ticket_export_app/core/tickets.py`

### 验证输出

1. 前30台完整序列。
2. 前100台A/B/C数量。
3. 前300台A/B/C数量。
4. 前600台A/B/C数量。
5. 全量A/B/C数量。
6. 实际最大连续同车型台数。
7. 修改前后总完成时间、整体节拍、等待和主要根因工位差异。

### 验收标准

1. 2:3:5的任意前缀分布明显比 `AABBBCCCCC` 整块重复更均匀。
2. 最大连续台数不超过用户设定。
3. 全量数量与目标比例一致。
4. 同一输入重复运行结果一致。
5. 不修改按数量投车模式的序列。
6. 两组真实数据输出修改前后差异；差异只能归因于投车序列变化，不得夹带排程容量、等待口径或UI修改。

## 8. 第3轮：R10 共用资源服务顺序确认与修正

### 问题

当前程序按车号逐车排完整条路径。对部分上游工位不作业、实际更早到达下游的车辆，仍可能被后到但车号较小的车辆压在后面。

真实数据到达顺序扫描显示：末端共用小跑道存在大量“后车更早完成上游有效作业，但仍按车号后服务”的记录。按比例场景相邻服务次序中有184组，按数量场景有293组；最大上游完成时间领先分别为321秒和387.5秒。

### 目标

先确认现场汇流规则，再让共用或单设备资源严格执行该规则。只有现场确认“先到先服务”时，才改为按车辆实际到达时间提供服务；如果现场要求保持原车序，则保留车号顺序，并把它明确记录为业务约束。

### 实施规则

1. 现场确认“先到先服务”时，排队顺序为“实际到达时间优先”。
2. 在“先到先服务”规则下，到达时间相同时车号较小者优先，保证结果稳定可复算。
3. 保持现有 `schedule()` 输入和返回数据结构。
4. 不修改车型投车序列。
5. 不修改R2已确认的节点容量规则。
6. 不修改R9已确认的投车序列规则。
7. 本轮不预设迁移排程主体；实际修改文件在现场规则确认和代码影响分析后确定，不借本轮重构无关排程结构。
8. 本轮实施前必须确认现场汇流规则：若现场严格保持原车序，则按车号顺序是业务约束；若共用工位按实际到达接车，则执行实际到达规则。确认结果写入实施过程文档。

### 主要修改文件

- 以现场规则确认后的影响分析为准；当前候选为 `ticket_export_app/core/tickets.py`
- 不因本轮任务改动无关的 `schedule_v2.py`

### 最小场景

X长工位：A车100秒、B车不作业；下游Y为共用单设备、加工10秒。

Car#1为A，Car#2为B。该场景用于确认现场规则，不在规则确认前预设唯一正确答案：

- 若现场先到先服务，Car#2在0秒到达Y，应在0秒开始、10秒完成；Car#1在100秒到达Y，应在100秒开始。
- 若现场严格保持原车序，Car#2需要等待Car#1；该等待属于业务规则，不得被误报为算法错误。

### 验收标准

1. 资源严格按现场确认的汇流规则服务。
2. 若现场规则为先到先服务，较早到达的车辆不会因为车号较大被凭空增加等待。
3. 同一资源无加工和占用重叠。
4. R2节点容量规则继续成立。
5. R9投车序列规则继续成立。
6. 真实小跑道184组/293组顺序差异必须逐项重算，并区分“按现场规则修正”与“只是算法顺序变化”。
7. 修改后的完整车辆时间链仍可逐项复算，正加工容量超限和同槽重叠均为0。

## 9. 第4轮：R1 等待根因正确归位

### 问题

当前程序可能把等待记到车辆记录中的下一行。下一行若是0工时节点，真实瓶颈就会被错记。

### 目标

等待根因追溯到下一个真正需要加工和资源的工位。

### 实施规则

1. 完工后等待沿车辆路径查找下一个工时大于0的有效作业工位。
2. 0工时节点只记录车辆停留位置，不作为能力瓶颈根因。
3. 删除“根因工位必须用原始工时直接超过目标节拍”的旧过滤。
4. 本轮只改归因，不改变任何车辆开始、加工完成、离开和下线时间。

### 主要修改文件

- `ticket_export_app/core/analysis.py`
- UI实时提示只同步读取统一分析结果

### 验收标准

1. ST11上的等待不再错误归到L2++。
2. 根因能够追到小跑道等真正下游工位。
3. 修改前后所有车辆时间链完全一致。

## 10. 第5轮：原始等待与岗位能力瓶颈等待分离

### 目标

原始时间链和业务瓶颈指标分别保存，不再混为一个累计阻塞数字。

### 最终口径

1. 原始流转等待：首工位投入等待 + 所有完工后等待下游接收。
2. 岗位能力瓶颈等待：只统计根因工位能力节拍超过目标节拍所造成的实际等待。
3. 工位能力节拍 = 当前车型工时 ÷ 有效设备能力。
4. 0工时节点不参加能力判断，等待原因继续追到下一个有效作业工位。
5. 加工时间超目标属于工位能力风险，不属于等待工时。

### 新增分析字段

- `total_raw_flow_wait`：累计原始流转等待；
- `total_bottleneck_wait`：累计岗位能力瓶颈等待；
- 每个工位的原始等待；
- 每个根因工位的瓶颈等待。

现有字段暂时保留兼容，本轮不切换模型结果卡片。

### 主要修改文件

- `ticket_export_app/core/analysis.py`

### 验收标准

1. 目标58秒、双设备：100秒和116秒工时均不超过能力节拍。
2. 两车之间16秒加工时长差不算等待。
3. 能力满足目标时产生的16秒同步停留保留在原始等待，但不进入瓶颈等待。
4. 单设备116秒场景进入瓶颈等待。
5. 原始等待可由车辆过程记录逐项复算。

## 11. 第6轮：模型结果切换为累计瓶颈阻塞

### 目标

模型结果区和风险提示使用最终业务口径，原始等待仍保留用于调查。

### 实施内容

1. “累计阻塞”卡片使用累计岗位能力瓶颈等待。
2. 阻塞工程使用统一根因分析结果。
3. 原始流转等待保留在车辆日志或调查信息中。
4. 结果说明注明：累计阻塞是多车多工位逐项累加，不能加到总完成时间。
5. 不修改下线车辆、整体节拍和总完成时间。

### 主要修改文件

- `ticket_export_app/ui/export_ticket_window.py`

### 验收标准

1. 卡片、风险提示、结果说明使用同一瓶颈等待数值。
2. 原始等待仍可查询。
3. 总完成时间和下线时间链不变。

## 12. 第7轮：达标口径统一

### 目标

车辆日志、模型结果和分析模块使用同一套达标判断。

### 唯一口径

下线车辆经过的所有工时大于0的工位，其“车型工时÷有效设备能力”均不超过目标节拍，则该车能力满足目标。

工时为0的节点保留流转记录，但不参加判断。

### 实施内容

1. 删除车辆日志中的OK/NG式最终判定表达。
2. 改为“能力满足”或列出“能力超过目标的工位”。
3. 模型结果、日志和分析结果调用同一个判断函数或同一份分析结果。

### 验收标准

- 随机20台及全部车辆在三处判断完全一致。

## 13. 第8轮：业务术语中文化

### 目标

用户不需要理解英文缩写即可看懂车辆日志和结果说明。

### 中文名称

| 内部字段 | 用户显示 |
|---|---|
| IN | 实际投车时间 |
| OUT | 下线完成时间 |
| WAIT | 等待工时 |
| FLOW | 单车贯通时间 |
| segment | 工位过程记录 |
| gap | 相邻下线间隔 |
| launch_wait | 首工位投入等待 |
| block_wait | 完工后等待下游接收 |

英文内部字段可保留，不修改数据接口。

### 主要修改文件

- `ticket_export_app/ui/export_ticket_window.py`
- 对应用户文档

### 验收标准

- 主界面、车辆日志和说明弹窗不依赖英文缩写也能理解。

## 14. 第9轮：实时节拍口径标注

### 目标

让使用者清楚实时节拍和整体节拍是两个不同观察范围。

### 实施内容

1. 实时节拍明确为“最近最多5个相邻下线间隔平均”。
2. 整体节拍继续为分析窗口内全部已下线车辆的平均间隔。
3. 不增加新算法和切换模式。

### 验收标准

- 1340秒和1830秒的实时节拍都可由对应车辆下线完成时间复算。

## 15. 第10轮：过程值与窗口终值分离

### 目标

同一画面明确区分最终统计结果和当前播放时刻状态。

### 实施内容

1. 模型结果区标注“分析窗口终值”。
2. 仿真栏标注“当前播放时刻”。
3. 顶部和底部等待提示使用同一分析数据源，仅通过时间范围区分。

### 验收标准

1. 不同播放时刻下，窗口终值保持不变。
2. 过程值随时间变化且能从车辆过程记录解释。
3. 不再出现没有时间口径说明的两套阻塞列表。

## 16. 第11轮：理论投车与仿真缓冲车辆披露

### 目标

让用户知道按比例模式中哪些是目标批次，哪些只是为了观察窗口尾部趋势而额外生成。

### 实施内容

1. 显示理论投车台数。
2. 单独显示仿真缓冲车辆数量。
3. 批次完成和超出时间只使用理论台数。
4. 缓冲车辆不进入目标批次完成判断。

### 验收标准

- 1150分钟、58秒场景显示理论1190台和仿真缓冲50台。

## 17. 第12轮：车辆日志筛选和中文导出

### 目标

提高一千台以上真实日志的查询和复核效率。

### 实施内容

1. 使用现有结构化车辆数据展示表格。
2. 支持车号、车型、工位筛选。
3. 支持仅查看存在岗位能力瓶颈等待的车辆。
4. 支持导出当前筛选结果为CSV。
5. 不重新运行排程，不修改模型结果。

### 验收标准

1. 可直接定位Car#11、Car#12和最后一台下线车辆。
2. 可单独列出瓶颈等待车辆及根因工位。
3. 导出内容与当前筛选结果一致。

## 18. 第13轮：岗位矩阵模板导入导出

### 目标

减少逐行录入工作，并防止错误模板直接覆盖当前数据。

### 模板字段

- 模板版本；
- 序号；
- 工程名称；
- 设备数量；
- 所属线别；
- 岗位设备；
- A/B/C工时。

### 实施规则

1. 导入前显示差异预览。
2. 校验失败时不修改当前矩阵。
3. 校验序号、必填字段、设备数、线别、工时、重复工程和行顺序。
4. 工时0统一解释为“经过但不作业”，不得再显示“跳过”。
5. 保留导入前矩阵快照，允许取消导入。

### 验收标准

1. 合法模板与人工录入生成相同岗位定义。
2. 错误模板不会覆盖现有矩阵。
3. 导出后重新导入结果一致。

## 19. 第14轮：排程死代码隔离

### 目标

隔离当前UI不可达的Zone/gate旧逻辑，降低后续维护误触风险。

### 实施规则

1. 不改变当前有效排程路径。
2. 不与任何功能修改合并。
3. 修改前后两组真实数据必须逐车一致。

### 验收标准

- 任意车辆的车型、开始、完成、离开和下线时间均不得变化。

## 20. 暂缓项目

以下项目需要单独确认业务定义，不在当前实施轮次中自动开展：

1. 动态达标指标：需先明确按计划下线时刻、相邻下线间隔还是窗口完成率判断。
2. 投车节拍与目标节拍解耦。
3. 设备数量大于2、单线多设备和用户可配置缓冲容量；本期R2只使用已确认的每工位每线1台流转车位规则。
4. 自动滚动画布、排队位次和进度条精度分别作为独立体验任务。

工位卡片内部的橙色“还有X台”不再作为独立改善任务。R2完成后，同一槽位仍出现多台车辆即视为异常，必须回到容量、线别或动画映射继续排查。

下方“加工中车辆”摘要中的“还有X台”属于全线列表折叠，不代表同一工位容纳多台车辆。当前该摘要存在二次压缩导致数量显示错误的问题，作为独立UI任务登记，不与R2核心排程修改合并。

在R2完成前，不得通过隐藏工位卡片的“还有X台”来掩盖同槽问题；必须先保证核心排程同槽重叠为0。
