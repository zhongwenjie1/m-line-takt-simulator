# 第16轮 全程序真实性与易懂性审核报告

## 1. 审核范围

本轮只读检查以下内容，不修改程序逻辑：

- 参数与岗位录入；
- 排列冻结、岗位矩阵导入导出；
- 分析与模型结果；
- 仿真回放与近期节拍；
- 车辆明细、等待真因与CSV导出；
- 单工程组合票；
- 精简排程分析报告式样；
- 使用说明、发布说明和历轮验证脚本。

审核基线：`main@b0e3eec`。

## 2. 总结论

当前核心排程在已定版规则下通过第1轮及第3至14轮专项验证。真实数量场景仍满足：

- A276 / B306 / C566，共1148台；
- 首台下线完成时间1423.5s；
- 最晚下线完成时间76152.5s；
- 整体节拍65.1517s/台；
- 累计实际等待239975.5s；
- 累计节拍外等待5430.0s；
- 同一资源加工重叠0次；
- 车位占用重叠0次；
- 共用/单设备先到先服务违规0次；
- 首工程投车越序0次。

但界面统计边界发现两项可复现的P0问题。在修复前，短分析窗口或分析后又修改参数的结果不能作为对外数据。

## 3. P0问题

### P0-1 比例模式零下线窗口错误回退为全部车辆

位置：`ui/export_ticket_window.py`中`_build_realtime_model_result()`。

当前逻辑先筛选`下线完成时间 <= 分析窗口`的车辆；当筛选结果为0台时，又把统计对象替换为全部仿真车辆。

真实复现：

- 比例A:B=4:2；
- 分析时间1分钟，即60s；
- 实际窗口下线0台；
- 仿真生成52台；
- 界面错误显示下线52台；
- 结果说明错误显示“第52台下线完成时间7260s <= 60s”。

核心分析结果中的`actual_output_count_in_window`为0，错误只发生在UI模型结果重建层。

影响：下线车辆、能力满足车辆、能力满足比例、整体节拍、等待指标和说明文字会同时使用错误统计范围。

### P0-2 修改输入后仍保留旧排程，并可能混合新参数重新显示

位置：`ui/export_ticket_window.py`的输入信号连接、`do_analyze()`和`_update_realtime_model_result()`。

当前输入变化不会清空`last_schedule_rows`和`last_analysis`。部分输入变化后，播放、重置或结果刷新会读取旧排程rows，同时读取界面上的新目标节拍或新模式。

真实复现：

1. 目标节拍118s分析后，排程完成时间1082.0s、整体节拍174.4s、累计实际等待82.0s。
2. 不重新分析，只把目标节拍改为58s并刷新结果。
3. 界面显示目标节拍58s，但仍使用旧排程：完成时间1082.0s、整体节拍174.4s、累计实际等待82.0s。
4. 按58s重新分析后的真实结果为：完成时间940.0s、整体节拍146.0s、累计实际等待510.0s、累计节拍外等待98.0s。

影响：界面可出现“输入是新参数、车辆时间链是旧参数”的混合结果，且车辆明细和导出可能与主界面不一致。

## 4. P1问题

### P1-1 最早的真实数据基线脚本仍使用旧结果

`verify_v29_real_data_baseline.py`仍断言比例窗口1046台、整体节拍65.1928s、累计等待243807.0s等旧值，当前执行失败。第1轮及第3至14轮专项脚本验证的是修复后的新结果。

风险：维护人员可能把正确的新结果误认为回归失败，或为通过旧脚本而反向破坏新逻辑。

> **【审查建议 2026-06-15｜P1-1】**
> 第16C 轮是后续所有轮回归依据，列为最前置 P0（阻塞性）。建议同时为每个权威指标生成 SHA-256 锁断言（如 rows 哈希、首末下线时间、累计等待、节拍外等待），任何意外动到排程都会被回归立即撞挂，避免再发生"旧脚本旧值漂移"。
> 保留旧脚本为"历史调查证据"快照，加状态标识，但不再作为通过条件。

### P1-2 数量模式的“结果说明”错误套用分析时间

数量模式没有分析时间窗口，但说明缓存会把隐藏的`spn_total_cars`值当成分钟。

真实复现：数量模式共6台，结果说明显示“第6台1082s <= 分析时间60分钟（3600s）”。数量模式应说明目标批次，而不是分析时间。

> **【审查建议 2026-06-15｜P1-2】**
> 建议在第16D 轮同时建立"**统计范围措辞表**"（项目内单一文本源），把"按目标批次共N台统计 / 最后完成车辆Car#X、X秒下线 / 按分析时间N分钟（X秒）"等措辞固化在常量里，主界面、结果说明、报告、使用说明全部从同一份引用。否则后面 16F 报告、16I 文档每改一处都得同步，又会再次漂移。

### P1-3 达标车辆和达标率容易被理解成排程表现

当前达标车辆实际表示“车型在所有正工时工程上的静态能力均不超目标”。真实数量场景为276/1148，基本等于A车型数量；比例窗口为206/1024，基本等于A车型占比。

该指标对排序、等待和下线节奏不敏感，名称容易被理解为“按时下线”或“本次排程达标”。

> **【审查建议 2026-06-15｜P1-3】**
> 同意从主界面移出。建议 16G 主界面做减法时，把"达标车辆/达标率"统一更名为"**车型能力满足（静态）**"放进分析报告的"能力观察"区，并保留底层判断不删，仅调整展示层级。主界面不再显示。

### P1-4 主界面信息层级过重

模型结果区同时显示：

- 5张结果卡片；
- 累计实际等待；
- 风险提示；
- 等待真因前三项；
- 结果说明入口。

其中“累计节拍外等待、节拍外等待发生工程、等待真因、超节拍工程”需要用户理解四个层级，主界面已接近调查页面而非结果总览。

> **【审查建议 2026-06-15｜P1-4】**
> 建议把 16F（接入报告）与 16G（主界面做减法）顺序对调为 **16G → 16F**：先把主界面砍出"次级摘要"边界，报告才能精确知道自己要补什么；否则报告做完后主界面一砍，报告又得回头加东西。
> 主界面四张主卡片建议固化为：① 下线车辆 ②（数量：批次总完成时刻 / 比例：窗口末台下线时刻） ③ 整体节拍 ④ 累计节拍外等待。次级摘要只保留：累计实际等待 + 主要节拍外等待位置最多3项 + 超节拍工程简表。等待真因长链放车辆明细。

### P1-5 结果说明、车辆明细和分析报告职责重叠

结果说明重复解释卡片；车辆明细同时承担查询、技术字段解释和等待真因；精简报告又包含结果、人工复算和口径说明。三个载体缺少明确主次。

> **【审查建议 2026-06-15｜P1-5 / 车辆明细 vs 报告 职责钉死】**
> 判断原则一句话：**车辆明细 = 在线查询工具；分析报告 = 离线复核证据**。两者读取同一份 rows/analysis，差别在于面向场景和是否冻结。具体边界如下表：
>
> | 项目 | 车辆明细（主程序内） | 分析报告（Excel 离线） |
> |---|---|---|
> | 场景 | 此刻查某台车 | 批次留档/复核/交付 |
> | 粒度 | 一车一行+可展开工程明细 | 一车一行（全量）+ 完整工程记录 |
> | 筛选 | 是，按车号/车型/工程/等待类型 | 否，固定全量 |
> | 冻结时点 | 跟随当前分析 | 导出时点快照（含时间戳/哈希） |
> | 等待真因 | **唯一**交互式追溯入口（点车号展开） | 全批次等待事件固定表 |
> | 能力满足判断 | 仅查询用 | 进"能力观察"区供复核 |
> | 导出 | 仅当前筛选 CSV | 完整 Excel（含口径说明 sheet） |
> | 完整口径说明 | 否（只解释筛选字段） | **唯一**载体 |
>
> 16H 实施时强制三条：① 车辆明细顶部不再放任何长篇口径说明；② 不在车辆明细内开第二个"导出报告"入口；③ 所有"完整口径说明"的提示一律指向报告。

### P1-6 现行使用说明与程序定版口径不同步

使用说明仍包含：

- “工时0可理解为跳过工位”；
- “累计阻塞”；
- “车辆明细 / 调试日志”；
- release文档仍写第5张卡片为累计阻塞。

当前程序已使用“经过但不作业”“累计实际等待/累计节拍外等待”“能力满足/能力超目标”。

> **【审查建议 2026-06-15｜P1-6】**
> 建议 16I 实施时加一条硬约束：**使用说明只写规则，不写本次结果数值**（如不写 1148 台、76152.5s 这类）。结果数值只属于 release/分析报告。否则使用说明每次跑完都得改一次，又会过期。
> 同时建议为对外术语做"**冒泡测试**"：用脚本扫主程序+报告+使用说明+release 文档里的中文字符串，出现一次"瓶颈阻塞"、"根因工程"、"下一站"等已废术语就算回归失败，杜绝口径漂移再次发生。

### P1-7 全部参与车型工时为0的工程被禁止录入

`core/input_parser.py`仍提示“参与投车的车型不能全部跳过该岗位”。这与当前业务语义“0工时仍经过节点但不作业”不一致。

如果现场矩阵允许某工程对当前全部车型都只作为流转节点，当前程序无法录入。

> **【审查建议 2026-06-15｜P1-7 范围窄化】**
> 业务上不存在“全程不作业的车型”，因此“全 0 工时投车”不是真实场景；但 P1-7 实际要修的不是这种极端，而是**当本次只投部分车型时**，某工程对“当前参与车型”恰好全部为 0 工时（仍对未投车型有作业）的合法用例无法录入。
> 建议把第16E轮的目标从“允许全 0 工时工程”窄化为“**取消‘参与车型不能全部跳过该岗位’这条校验**”，错误文案改为友好提示并允许继续；其余非负、数值格式、设备数与线别校验保持不变。
> 若现场确认这种矩阵永远不会出现（包括未来分批投车场景），第16E轮可直接取消。建议先保留，工作量与风险都小。

### P1-8 小屏仍依赖内部滚动，窗口最小高度偏大

窗口最小尺寸为1080x700。离屏验证结果：

- 1080x700：结果页垂直滚动范围162px；
- 1366x768：垂直滚动范围94px；
- 1440x900：无需滚动。

当前画布与事件区未重叠，但13寸Windows在系统缩放后可用逻辑高度可能低于700px，窗口底部和入口仍有不可见风险。

> **【审查建议 2026-06-15｜P1-8】**
> codex 计划写到最小 900×520，偏激进。建议最小定为 **1024×600**，理由：① 13 寸 Windows 125% 缩放后逻辑高度约 614px，1024×600 仍可覆盖；② 离屏测试 1080×700 已能用，900×520 实际未测，过窄时 PySide6 部分按钮会叠到不可点状态。
> 字体不缩、动画坐标不改，参数页和结果页允许垂直滚动。

### P1-9 真实规模分析在UI线程同步运行

本机真实数量场景核心排程约0.83s，分析约2.05s；比例场景约0.91s和2.35s。`do_analyze()`同步执行，并继续构建动画和结果。运行期间无忙碌状态，低配置Windows上可能被理解为未响应或重复点击。

> **【审查建议 2026-06-15｜P1-9】**
> 同意异步化。补一条 codex 计划没写的：**增加"取消正在进行的分析"按钮**。1148 台约 3s 还能忍，但更大规模（5000 台或更复杂矩阵）会到 15s+，没有取消按钮用户会强制关窗口。Worker 内查响应位，UI 收到取消后恢复输入区和按钮即可。
> 配合使用分析请求编号 + 输入签名，已过期的 Worker 结果直接丢弃，不写回界面。

### P1-10 排列冻结不能跨电脑复现

冻结序列只保存在当前内存，改变输入后失效，没有冻结文件导出/导入。相同参数和当前算法仍是确定性的，但不同版本、不同规则或人工确认的序列无法通过文件留档复现。

> **【审查建议 2026-06-15｜P1-10】**
> codex 列的 schema 字段完整，补一条：**"程序版本"作为硬校验**（不一致直接拒绝导入，不是警告）。冻结的本质是"在某个程序版本下确定的序列"，版本变了规则之外的排程也可能变。
> 文件扩展名 `*.mline-sequence.json` 可，但建议导入失败时弹出**逐项差异表**（哪个字段不一致、当前值是多少、文件里是多少），方便现场人员定位，不静默覆盖任何参数。

## 7. UI信息架构结论

### 主界面模型结果

主界面只承担“看结果、看是否需要进一步调查”。候选保留信息：

1. 下线车辆；
2. 批次总完成时刻（数量模式）或分析窗口末台下线时间（比例模式）；
3. 整体节拍；
4. 累计节拍外等待；
5. 累计实际等待作为次级摘要，而不是主判断卡片。

“达标车辆/达标率”不适合作为排程表现主卡片。若保留，名称必须明确为静态工程能力观察，并移到报告或次级区域。

### 风险与等待

主界面只保留：

- 主要节拍外等待位置；
- 超节拍工程。

完整的“等待发生工程 -> 直接阻挡 -> 等待真因”链放在车辆明细或分析报告，不在主界面展开前三条长句。

### 车辆明细

车辆明细承担现场查询：按车号、车型、工程和等待类型定位，并导出当前筛选CSV。窗口名称不再使用“调试日志”。

### 分析报告

完整Excel承担人工复算、汇总口径和批次留档。它是完整证据载体，不与车辆明细争夺“当前筛选查询”的职责。

### 结果说明

现有结果说明与报告口径页重复。若分析报告正式接入，现有长弹窗没有独立保留价值。主界面只需短说明或报告入口，详细公式进入报告。

7. **第16G轮：接入正式分析报告导出。** 只读取已有rows与analysis生成Excel，不重新排程。
8. **第16H轮：车辆明细职责收口。** 只改名称、说明和入口，不改时间链。
9. **第16I轮：同步用户文档。** 只更新使用说明、发布说明和历史文档状态标识。
10. **第16J轮：小屏与忙碌状态。** 分开执行窗口尺寸优化和分析进行中状态，不合并修改。
11. **后续轮：冻结序列导出/导入。** 只处理复现文件格式，不改变排序算法。

## 8A. 综合优先级建议（新增 2026-06-15）

> 本节为新增，不修改第 8 节原清单。第 8 节为 codex 初版边界定义；本节为审查后的综合排序与依赖关系建议。

### 8A.1 优先级与依赖

```
P0 阻塞性（必须最先做，后续轮回归依据）
  16C  权威基线 + SHA-256 锁
  16D  统计范围措辞表（数量/比例统一文本源）

P1 主线（按下列顺序，前一轮为后一轮前提）
  16E  范围窄化为"取消过严校验"           ← 独立小项，可并入早期任一空档
  16G  主界面做减法                       ← ※ 与 codex 顺序对调
  16F  接入分析报告                       ← 砍主界面后才知道报告补什么
  16H  车辆明细去重收口                   ← 三处口径在此统一
  16I  同步使用说明与发布文档             ← 三处对外口径定版后再做

P2 收尾（顺序无强依赖，可并行）
  16J  小屏布局（最小 1024×600）
  16K  分析异步化 + 取消按钮
  16L  冻结序列导出/导入（程序版本硬校验）
```

### 8A.2 与 codex 原计划的关键差异

| 项 | codex 原计划 | 本节建议 | 原因 |
|---|---|---|---|
| 16E 范围 | 允许全 0 工时工程 | 取消"参与车型不能全部跳过该岗位"校验 | 真实场景不存在全 0 投车；问题在校验过严，应窄化 |
| 16F / 16G 顺序 | 先报告（16F）后主界面（16G） | **对调为 16G → 16F** | 先定主界面砍出哪些进次级，报告才能精确补缺；否则报告需返工 |
| 16J 最小窗口 | 900×520 | **1024×600** | 13 寸 Windows 125% 缩放后逻辑高度约 614px；900×520 未实测，过窄 PySide6 控件会叠到不可点 |
| 16K 异步化 | 仅禁用按钮 + 忙碌状态 | **追加"取消正在进行的分析"按钮** | 更大规模分析时间会到 15s+，无取消则用户会强制关窗 |
| 16L 冻结导入 | 校验差异列出后拒绝冻结 | **同样拒绝，但"程序版本"作为硬校验**，并弹出逐项差异表 | 程序版本变了规则之外的排程也会变；差异表便于现场定位 |

### 8A.3 全程回归追加项

在 codex 原回归要求基础上追加两项：

1. **基线 SHA-256 锁**：每轮跑完 16C 建立的权威指标 SHA-256 锁断言；任何意外触动排程立即被回归撞挂。
2. **对外口径冒泡测试**：脚本扫主程序+报告+使用说明+release 文档里的中文字符串，出现"瓶颈阻塞 / 根因工程 / 下一站 / 调试日志 / 累计阻塞 / 工时0可理解为跳过工位"等已废术语就算回归失败，杜绝口径再次漂移。

### 8A.4 单一文本源原则

为彻底解决"同一术语多处定义"导致的反复漂移，建议本轮新增两个**单一文本源**常量文件（或同一文件的两个常量块），所有 UI、说明、报告、release 文档都引用同一份：

- **统计范围措辞**（数量"按目标批次共 N 台统计"、比例"按分析时间 N 分钟（X 秒）"）
- **核心业务术语表**（实际等待 / 累计节拍外等待 / 可接纳上限 / 下一工程（路线，非根因） / 超节拍工程 / 经过但不作业）

后续任何文档或界面新增显示，先到这两份常量里查表；缺词先补常量，再写显示。

## 9. 是否进入修改阶段

不自动进入修改阶段。需要用户先确认P0-1、P0-2以及后续单任务顺序。
